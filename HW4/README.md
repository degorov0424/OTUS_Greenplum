# Уровни изоляции

## 1.Неповторяемое чтение

Была проведена проверка неповторяемого чтения с разными уровнями изоляции: READ COMMITTED и REPEATABLE READ
Использовались скрипты `isolation_session_1.sql` и `isolation_session_2.sql`.
READ COMMITTED - при повторном чтении данных в рамках одной транзакции их значение оказывается измененным другой транзакцией
REPEATABLE READ - неповторяющееся чтение исключено

---

## 2.Обновление одной строки в разных сессиях

Была проведена проверка обновления одной строки в двух транзакциях. 
Запускалась сессия 1, после чего в сессии 2 выполнялся update записи, затем выполнялся update этой же записи в сессии 1.
В результате:
READ COMMITTED - обновление завершилось без ошибок
REPEATABLE READ - обновление упало с ошибкой SQL Error [0A000]: ERROR: updates on append-only tables are not supported in serializable transactions

Так же был проведен аналогичный тест, но в сессии 2 выполнялся update без commit, для проверки блокировки. 
В результате, обновление в сессии 1 зависало. 
После выполнения commit в сессии 2, обновление в сессии 1 завершалось либо успешно (READ COMMITTED), либо с ошибкой (REPEATABLE READ) 

---

## 3.Грязное чтение

Т.к в Greenplum READ UNCOMMITTED работает так же как и READ COMMITTED, чтение не зафиксированных данных, измененных в другой транзакции, невозможно с уровнем изоляции READ UNCOMMITTED

# Блокировки

## 1.ACCESS SHARE и ACCESS EXCLUSIVE

В скрипте `lock_session_1.sql` запустил запрос в транзакции 1, но не завершил транзакцию. 
Скрипт `locks.sql` показал блокировку AccessShareLock на таблице orders 
В скрипте `lock_session_2.sql` запустил транзакцию 1 с `ALTER TABLE tds2.orders`. Транзакция зависла т.к ACCESS SHARE конфликтует с ACCESS EXCLUSIVE. 
После завершения транзакции в сессии 1, завершается и сессия 2.   

Если запустить в обратном порядке, т.е сначала `ALTER TABLE tds2.orders`, то `SELECT * FROM tds2.orders` так же блокируется, т.к ACCESS EXCLUSIVE конфликтует со всеми операциями, в т.ч. с SELECT

## 2. ROW EXCLUSIVE, EXCLUSIVE и ACCESS SHARE
В скрипте `lock_session_1.sql` запустил вставку в транзакции 2, но не завершил транзакцию.
Скрипт `locks.sql` показал блокировку RowExclusiveLock на таблице orders.
В скрипте `lock_session_2.sql` запустил первый запрос в транзакции 2. SELECT выполнился. 
Но следующий UPDATE уже заблокировался, т.к ROW EXCLUSIVE конфликтует с EXCLUSIVE, но не с ACCESS SHARE.
